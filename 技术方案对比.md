# Superpowers vs Superharness — 实际使用差异对比

## 一句话总结

**Superpowers** 是 Jesse Vincent 写的开源"通用驾照"——装完就能用，适配 8 个平台，14 个技能覆盖完整开发流程。
**Superharness** 是你自己写的"项目内部规章"——只在 Claude Code 上跑，7 个技能，砍掉了通用性，换来了更简单直接的体验。

---

## 1. 启动机制

| | Superpowers | Superharness |
|---|---|---|
| **注入脚本** | Bash（跨平台），自动识别当前运行在哪个平台上 | PowerShell（仅 Windows） |
| **注入内容** | `using-superpowers/SKILL.md` 全文（~200 行，包含完整的技能触发规则、Red Flags 表、技能优先级等） | `HARNESS.md`（~50 行，精简版规则） |
| **触发规则** | "哪怕只有 1% 可能，做任何事情之前都要检查技能"——包括回答问题 | "做任何工程任务前检查技能"——范围更窄 |
| **旧版迁移** | 会检查并提醒你把 `~/.config/superpowers/skills` 搬到 `~/.claude/skills` | 无 |
| **平台兼容** | 一套代码多平台——通过 `CLAUDE_PLUGIN_ROOT` / `CURSOR_PLUGIN_ROOT` / `COPILOT_CLI` 等环境变量判断当前平台，输出对应格式的 JSON | 只认 Claude Code，PowerShell 硬编码 |

**实际体验：** Superpowers 启动时 AI 脑子里被塞进一整套"行为规范手册"，Superharness 只塞进去一张"速查卡"。

## 2. 技能数量与覆盖范围

| 功能 | Superpowers | Superharness |
|---|---|---|
| **入口技能** | `using-superpowers`（技能系统的总开关） | HARNESS.md 兼做入口 |
| **需求设计** | `brainstorming`（自动触发，9 步流程，视觉伴侣需用户同意才开启） | `brainstorm`（**仅手动触发** `/superharness:brainstorm`，脑图强制开启，可降级） |
| **实施计划** | `writing-plans` → 保存到 `docs/superpowers/specs/` | `writing-plans` → 保存到 `superharness/plans/` |
| **任务执行** | `executing-plans`（批次+检查点）+ `subagent-driven-development`（子代理两道审查）+ `dispatching-parallel-agents`（并行） | `go`（一个命令驱动 5 个阶段：理解→计划→实现→验证→审查→报告） |
| **TDD** | `test-driven-development`（含反模式参考） | `test-driven-development`（反理性化表更激进，多了 2 条常见借口） |
| **调试** | `systematic-debugging`（4 阶段根因分析 + 3 个子技巧） | `systematic-debugging`（同样 4 阶段 + 同样 3 个子技巧，几乎完全一致） |
| **代码审查** | `requesting-code-review` + `receiving-code-review`（既要会提审查，也要会接审查） | 只有 `requesting-code-review`（只管提） |
| **收尾** | `finishing-a-development-branch`（合并/PR/保留/丢弃 + 清理 worktree） | 内嵌在 `go` Phase 5 的 Report 里（更轻量） |
| **隔离工作区** | `using-git-worktrees`（在流程中自动触发） | `using-git-worktrees`（有技能但不在 `go` 主流程里） |
| **验证** | `verification-before-completion` | `verification-before-completion`（基本一致） |
| **写技能** | `writing-skills`（元技能——教你写新技能、对抗测试） | 无 |

**实际体验：** Superpowers 像一本百科全书，Superharness 像一本口袋手册——只留了天天用的那几页。

## 3. 头脑风暴（brainstorming）——差别最大的环节

| | Superpowers `brainstorming` | Superharness `brainstorm` |
|---|---|---|
| **触发方式** | **自动**——你只要说"做个 XX"，AI 就会被强制拉进 brainstorming | **手动**——你必须显式打 `/superharness:brainstorm <主题>` |
| **脑图/视觉** | "视觉伴侣"——可选，需要征得你同意才开。同意后 AI 自己判断每个问题用浏览器还是终端 | **强开脑图**——自动启动 Node 服务器，推一个实时思维导图到浏览器。Node 没装才降级纯文本 |
| **流程步骤** | 9 步（探索→问是否需要视觉→提问→方案→设计→写文档→自审→用户审→转实施） | 6 阶段（启动脑图→探索→提问→方案→设计→收尾），结构类似但更紧凑 |
| **设计文档位置** | `docs/superpowers/specs/YYYY-MM-DD-<主题>-design.md` | `superharness/specs/YYYY-MM-DD-<主题>.md` |
| **输出衔接** | 自动调用 `writing-plans` | 提示用户运行 `/superharness:go`，**不自动继续** |
| **反模式覆盖** | "这太简单不需要设计"——专门的反模式条目 | Red Flags 表（脑图挂了不阻塞、一次一个问题、批准后不顺手写代码） |

**实际体验：** 
- Superpowers：你说"做个 todo"，AI 说"等一下，先聊聊你到底想做什么"——自动的。
- Superharness：你说"做个 todo"，AI 可能直接开工。你得手动 `/superharness:brainstorm 做个 todo` 才会进入设计对话。但一旦进入，Superharness 直接给你开个脑图网页，你可以拖拽节点看设计结构。

## 4. 任务执行（go）——Superharness 的独有武器

Superpowers 把任务执行拆成了 3 个独立技能，用户/Agent 需要自己选：
- `executing-plans`：老老实实一个一个做，每批之间人工检查点
- `subagent-driven-development`：每个任务派一个子代理，两道审查
- `dispatching-parallel-agents`：不依赖的任务并行干

Superharness 只有一个 **`go`** 命令，一条龙：
```
/go superharness:go <目标>
```
自动走完 5 个阶段：
1. **Understand** — 理解目标，探代码，歧义大了才问一次
2. **Plan** — 3 步以上写计划，存 `superharness/plans/`
3. **Implement** — 严格 TDD，不允许跳过
4. **Verify** — 跑全量测试套件，不只看新增的
5. **Review** — 派审查子代理，严重/重要问题必须修
6. **Report** — 最终汇总报告（改了啥、证据、假设、遗留问题）

**实际体验：** Superharness 的 `go` 是一条"不归路"——按了回车它就自动驾驶，只在真正需要你拍板时才停。Superpowers 的流程给你更多中途介入的机会。

---

## 5. TDD 执行的激进程度

两边的 TDD 技能核心循环完全一致：RED → Verify RED → GREEN → Verify GREEN → REFACTOR → 循环。

但 **Superharness 版本更"凶"**：

| 对比项 | Superpowers | Superharness |
|---|---|---|
| **铁律措辞** | "NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST" | "NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST"（一致） |
| **反理性化条目** | 11 条常见借口 | **13 条**——多了"Tests after achieve the same purpose"、"Keep as reference or adapt existing code"、"Already spent X hours, deleting is wasteful"、"TDD is dogmatic, I'm being pragmatic" |
| **"为什么顺序重要"解释** | 有，较简短 | **长篇论述**——专门有一节"为什么顺序重要"，逐一驳斥 5 种常见偷懒借口 |
| **反模式参考** | 有 `testing-anti-patterns.md` | 同样有 `testing-anti-patterns.md` |

**实际体验：** Superharness 的 TDD 文档更像"写给你自己看的防偷懒指南"——知道自己的弱点，提前把每个借口的反驳都写好了。

---

## 6. 代码审查的完整度

Superpowers 有完整的审查闭环：
- `requesting-code-review`：提审查请求，按严重度（Critical / Important / Minor）分级
- `receiving-code-review`：**收到审查意见后怎么回应**——不能"表演式同意"、不能盲目照改、要技术验证

Superharness **只有 `requesting-code-review`**，没有"如何接受审查"的指引。

**实际体验：** 要是在 Superharness 下另一个 Agent 审查了你的代码，你没有技能来规范你的回应行为。在 Superpowers 下两边都有规矩。

## 7. 目录与文件组织

| | Superpowers | Superharness |
|---|---|---|
| **根目录** | 仓库根（`/`） | `.claude/superharness/` 下 |
| **技能位置** | `skills/<技能名>/SKILL.md` | `plugins/superharness/skills/<技能名>/SKILL.md` |
| **计划存储** | `docs/superpowers/specs/`（设计）、按执行技能的约定走 | `superharness/plans/`（计划）、`superharness/specs/`（设计） |
| **平台适配** | `.claude-plugin/`、`.codex-plugin/`、`.cursor-plugin/`、`.opencode/`、`gemini-extension.json` —— 每个平台独立清单 | 只有 `.claude-plugin/` |
| **测试** | 完整的 `tests/` 目录 | 无项目级测试 |
| **文档** | `docs/` 目录 + README + RELEASE-NOTES | 无独立文档 |

| | Superpowers | Superharness |
|---|---|---|

## 9. 核心哲学差异

两边底层的工程哲学是一致的：**TDD、系统化、证据优于声称、降低复杂度**。

但在"人机关系"上有微妙差异：

| | Superpowers | Superharness |
|---|---|---|
| **对人的称呼** | "your human partner"（你的人类搭档） | "your human partner"（一致） |
| **自主程度** | 中等——brainstorming 需要你来来回回确认，执行计划每批有检查点 | 更高——`go` 命令强调"autonomously"，只在真正需要你拍板时才停 |
| **设计阶段** | 必须走完 brainstorming 的设计确认才能写代码 | `go` 里把"理解"放在 Phase 0，只有歧义大到影响架构才问你，否则"选最合理的解释并记下假设" |
| **反模式表** | 散落在各技能里 | 集中在 HARNESS.md + 各技能里 |

## 10. 怎么选？

| 场景 | 推荐 |
|---|---|
| 你用多个 AI 编程工具（Claude Code + Cursor + Gemini CLI） | Superpowers（一套技能通吃） |
| 你只在 Claude Code 上干活 | 两个都可以，Superharness 更轻 |
| 你希望 AI **自动**进入需求讨论，不给你机会跳过设计 | Superpowers（brainstorming 自动触发） |
| 你希望自己控制节奏，需要时手动开脑图设计 | Superharness（brainstorm 手动，go 自动） |
| 你维护一个开源项目，希望贡献流程规范 | Superpowers |
| 你维护个人项目，想快速迭代 | Superharness（go 一条龙） |
| 你想自定义工程纪律规则 | Superharness（改 markdown 就行，不用过 PR） |
| 你需要完整的代码审查闭环（提审+接审） | Superpowers |
| 你是 Windows 用户 | Superharness（PowerShell 原生）或 Superpowers（Bash 也能跑） |
