# Superharness 技术方案文档

> 一句话：**Superharness 是给 Claude Code 装的一套"工程纪律外挂"。** 跑一条命令，
> 就能让 AI 在你的项目里按"先写测试 → 系统化调试 → 完成前必须验证 → 代码审查"的
> 规矩干活，并附带一个浏览器实时脑图来梳理需求。

本文用尽量简单的语言讲清楚：它**是什么、怎么装、运行时发生了什么、各部分怎么配合**，
并配有流程图。

---

## 1. 它解决什么问题

默认的 AI 编码助手容易"图快"：不写测试、猜着改 bug、嘴上说"修好了"却没真跑过。
Superharness 把一套硬性纪律以"插件 + 钩子 + 技能"的形式塞进项目，让 AI **每次会话
开始就被强制加载规则**，从而：

| 痛点 | Superharness 的约束 |
|------|---------------------|
| 不写测试 | **TDD 强制**：先写会失败的测试（红）→ 最小实现（绿）→ 重构 |
| 猜着改 bug | **系统化调试**：先定位根因，禁止"试一下"式补丁 |
| 假装完成 | **完成前验证**：必须贴出真实命令输出才能说"done" |
| 无计划乱干 | **先写计划**：3 步以上的任务先拆成 2–5 分钟的小步 |
| 不审查就合并 | **代码审查**：收尾前派子代理审 diff |

## 2. 核心概念（30 秒看懂）

```mermaid
flowchart LR
    CLI["命令行 superharness<br/>(一次性安装)"] -->|拷贝模板| PROJ[".claude/superharness<br/>本地插件市场"]
    PROJ -->|Claude Code 自动加载| PLUGIN["superharness 插件"]
    PLUGIN -->|SessionStart 钩子| RULES["每次会话注入<br/>HARNESS.md 规则"]
    PLUGIN -->|提供技能| SKILLS["/superharness:go<br/>/superharness:brainstorm<br/>+ 5 个核心技能"]
```

三层关系：
1. **CLI 安装器**（`bin` + `lib/install.ps1`）：只跑一次，把"模板"拷进你的项目。
2. **本地 marketplace 插件**（`.claude/superharness`）：Claude Code 启动时自动识别加载。
3. **钩子 + 技能**：钩子负责"每次都注入规则"，技能负责"具体怎么干活"。

## 4. 安装流程（运行 `superharness` 时发生了什么）

CLI 入口 `bin/superharness.cmd` 只是转发，真正逻辑在 `lib/install.ps1`。

```mermaid
flowchart TD
    START([用户在项目根目录运行 superharness]) --> ARGS{解析 --template / --stack}
    ARGS -->|非法组合| ERR["报错退出<br/>(如 --stack 缺 --template)"]
    ARGS -->|合法/无参| COPY["1. 拷贝 template → .claude/superharness<br/>(幂等覆盖)"]
    COPY --> STACKW{指定了 --template?}
    STACKW -->|是| WSTACK["写入选中的 STACK.md"]
    STACKW -->|否| RMSTACK["移除残留的 STACK.md"]
    WSTACK --> MERGE
    RMSTACK --> MERGE["2. 保留性合并 settings.json<br/>(extraKnownMarketplaces + enabledPlugins)"]
    MERGE --> LEGACY["3. 清理旧版 skills 目录"]
    LEGACY --> CMD["4. CLAUDE.md 写/替换 SUPERHARNESS 标记段<br/>(幂等，不重复追加)"]
    CMD --> DONE([打印后续步骤，退出 0])
```

**设计要点（为什么这么做）**：
- **幂等**：重复运行只覆盖更新，不破坏已有配置；`settings.json` 用"保留性合并"，
  只加自己的键，不动用户其它配置。
- **标记段**：`CLAUDE.md` 用 `<!-- SUPERHARNESS:BEGIN/END -->` 包裹，便于原地替换升级。
- **可选技术栈**：`--template=frontend|backend|fullstack`（配合 `--stack`）会把对应的
  栈指引拷成 `STACK.md`，供钩子注入。

## 5. 会话启动流程（每次打开 Claude Code）

插件的 **SessionStart 钩子**（`session-start.ps1`）保证规则"每次都在"。

```mermaid
flowchart TD
    OPEN([Claude Code 启动会话]) --> LOAD["从 .claude/superharness 自动加载插件"]
    LOAD --> HOOK["触发 SessionStart 钩子"]
    HOOK --> READ{HARNESS.md 存在?}
    READ -->|否| EXIT0["静默退出 0<br/>(钩子绝不阻塞会话)"]
    READ -->|是| WRAP["包成 EXTREMELY_IMPORTANT 块"]
    WRAP --> STACKQ{STACK.md 存在?}
    STACKQ -->|是| APPEND["追加技术栈指引"]
    STACKQ -->|否| OUT
    APPEND --> OUT["输出 additionalContext (JSON)"]
    OUT --> INJECT([规则注入本次会话上下文])
```

**容错设计**：钩子 `$ErrorActionPreference = 'SilentlyContinue'` 且永远 `exit 0`——
一个坏掉的钩子绝不能让用户无法开工。

## 6. `go` 技能：六阶段自主工作流

用户输入 `/superharness:go <任务目标>` 后，AI 按固定六步推进，每步对应一个子技能。

```mermaid
flowchart LR
    G([go 任务目标]) --> P1["① 理解<br/>探索代码 / 确认目标"]
    P1 --> P05["①.5 隔离<br/>using-git-worktrees<br/>默认建 worktree/分支"]
    P05 --> P2["② 计划<br/>writing-plans<br/>拆成 2-5 分钟 TDD 小步"]
    P2 --> P3["③ 实现<br/>subagent-driven-development<br/>每任务新子代理 · TDD"]
    P3 -->|遇到 bug| DBG["systematic-debugging<br/>定位根因"]
    DBG --> P3
    P3 --> P4["④ 验证<br/>verification-before-completion<br/>跑全套测试，贴真实输出"]
    P4 --> P5["⑤ 审查<br/>requesting-code-review<br/>子代理审 diff"]
    P5 -->|有严重问题| P3
    P5 -->|通过| DONE([收尾])
```

每个阶段是一个独立技能文件（`skills/<name>/SKILL.md`），核心内容移植并适配自
[obra/superpowers](https://github.com/obra/superpowers)。

### 6.1 隔离与子代理实现

- **Phase 0.5 隔离**：`using-git-worktrees` 在 git 项目里默认建 worktree/分支，让自主+自动提交的 `go` 改坏可干净回滚；非 git 或建失败则原地工作，绝不阻塞。
- **Phase 2 子代理实现**：多任务计划委托 `subagent-driven-development`，每个相互独立的任务派一个**全新实现子代理**（带完整任务文本+上下文，不读 plan 文件），主代理上下文只保留计划、协调与审查。逐任务只做自检，质量统一交给 Phase 4 的最终 `requesting-code-review`。
- **与 trace 的关系**：实现子代理**不写 trace 标记**；`outcome.json` 仍由主代理在 Phase 3 写、Stop 钩子按主代理轮次记录，因此**追踪粒度不变**。附带好处：trace 状态按 `cwd` 区分，不同 worktree 并发跑 `go` 天然不互踩。

## 7. 任务追踪与 `resume`：复现并修好失败的任务

`go` 跑任务时容易"一路向前"，一旦中途失败，重开会话就丢了上下文。这一层把
**每一轮需要用户介入的对话**持久化成"每任务一个、单行最小化的 JSON"追踪文件，并提供
`/superharness:resume` 入口：从追踪文件读回上下文，**由人确认后**走"复现 → 定位根因 →
改出问题的代码 → 验证"的闭环，把失败真正修好（而不是盲目重跑）。

### 7.1 混合式架构：hook 负责"记录"，Claude 负责"判定"

记录的**可靠性**交给 Claude Code 的 hook（自动触发，不依赖 Claude 记得做事）；
成败的**语义**交给 `go` 工作流写的一个小标记文件。两者分工明确：

| 谁 | 触发/时机 | 干什么 |
|----|-----------|--------|
| `UserPromptSubmit` hook | 每次用户提交 prompt | 暂存 `{ts, query}` 为"待结算的一轮" |
| `go` 工作流（Claude） | 任务开始 / 跑完测试 | 写 `task.json`（任务态）、`outcome.json`（成败标记） |
| `Stop` hook | Claude 交还控制权 | 合成该轮、追加进 `<task>.json`、消费临时标记 |

**作用范围限定**：只有存在任务态标记 `task.json`（即有 `go` 任务在跑）时才记录；
普通对话 / 非 go 会话的 `Stop` 一律 no-op，不产生游离文件。

### 7.2 单轮记录的数据流

```mermaid
flowchart TD
    U([用户提交 prompt]) --> UPS["UserPromptSubmit 钩子<br/>写 .state/&lt;sid&gt;.prompt.json = {ts, query}"]
    UPS --> WORK["Claude 工作 (go)"]
    WORK --> T1["任务开始: 写 .state/task.json<br/>{task_id, slug, goal, started_at}"]
    WORK --> T2["跑完测试: 写 .state/outcome.json<br/>{outcome, test_command, failing_tests?, task_status?}"]
    T1 --> STOP
    T2 --> STOP{Claude 交还控制权<br/>触发 Stop 钩子}
    STOP --> HASTASK{存在 task.json?}
    HASTASK -->|否| NOOP["no-op, exit 0<br/>(非 go 会话不记录)"]
    HASTASK -->|是| MERGE["合并 prompt.json + outcome.json<br/>分配轮号 n, 按规则裁剪"]
    MERGE --> APPEND["追加进 trace/&lt;task&gt;.json 的 rounds[]<br/>更新顶层 status / updated_at (原子写)"]
    APPEND --> CONSUME["删除 prompt.json / outcome.json (消费)"]
    CONSUME --> CLOSE{outcome 带 task_status?}
    CLOSE -->|是| FIN["写入顶层 status + 删除 task.json (收尾)"]
    CLOSE -->|否| END0([exit 0])
    FIN --> END0
```

**关键不变量**：`ts / query / n` 由 hook **无条件**提供——所以"每一轮都被记录"不依赖
Claude 的记忆；`outcome / failing_tests / test_command` 由 Claude 的标记**增强**，
缺失时该轮降级为 `in_progress`，绝不当作门槛。

### 7.3 成败判定（看 test case）与文件布局

| 该轮情况 | outcome | 记录内容 |
|---|---|---|
| 跑了测试且全绿 | `success` | `summary:"task completed"` + `test_command` |
| 跑了测试且有失败 | `failure` | `failing_tests`(名/文件/消息) + `query` + `ts` + `test_command` + `notes` |
| 该轮没跑测试（如澄清提问） | `in_progress` | `query` + `ts` + 一句话 `summary` |
| outcome 标记缺失 | `in_progress` | `query` + `ts`（hook 兜底） |

落在**目标项目**里（路径取自 hook 传入的 `cwd`，与 `superharness/plans/` 对齐）：

```
superharness/trace/
  2026-06-12-<slug>.json     # 交付物：每任务一个，单行最小化 JSON（保留在仓库，供 resume 读取）
  .state/                    # 临时态，已 gitignore
    task.json                #   go 任务开始时写：{task_id, slug, goal, started_at}
    <sid>.prompt.json        #   UserPromptSubmit 写：{ts, query}
    outcome.json             #   go 跑完测试后写：成败标记
```

> 失败轮**不**落整段对话原文——只记失败用例 + query + 关键重试信息，避免追踪文件膨胀。

### 7.4 `resume` 工作流

`/superharness:resume [slug]`（**仅手动触发**，`disable-model-invocation`，从不自我调用）：

```mermaid
flowchart TD
    R([/superharness:resume slug?]) --> P0["① 读取追踪<br/>按 slug/路径，或最新 status≠completed 的文件"]
    P0 --> NONE{有可恢复的追踪?}
    NONE -->|无| STOP0([告知无可恢复，停止])
    NONE -->|有| SUM["打印 goal / 各轮历史 / 最后一次 failing_tests + test_command"]
    SUM --> GATE{人工确认?}
    GATE -->|未确认| WAIT([硬门：等待确认，绝不自动重试])
    GATE -->|确认| BOOT["①′ 若 task.json 缺失则按原 task_id/slug/goal 重建<br/>(否则 Stop 钩子 no-op，复现不被记录)"]
    BOOT --> REPRO["② 复现 (RED)<br/>重跑 test_command，确认仍按记录原因失败"]
    REPRO --> GONE{已不再复现?}
    GONE -->|是| ASK([报告失败已消失，问是否继续原目标])
    GONE -->|否| ROOT["③ 定位根因<br/>superharness:systematic-debugging（禁止猜测打补丁）"]
    ROOT --> FIX["④ 改码 (TDD)<br/>失败用例即现成 RED → 最小改动转 GREEN"]
    FIX --> VERIFY["⑤ 验证<br/>verification-before-completion：跑全量测试，贴真实输出"]
    VERIFY --> GREEN{全绿?}
    GREEN -->|是| OKR["写 outcome task_status:completed<br/>追踪记一条 success 轮"]
    GREEN -->|否| FAILR["追踪记又一条 failure 轮<br/>(累积完整重试历史)"]
    FAILR --> ROOT
    OKR --> REPORT([⑥ 报告：根因 / 改动位置 / 验证输出 / 追踪状态])
```

**设计要点**：
- **不自动重试**：读回上下文后必须**先与人确认**才动代码——这是硬门。
- **修的是代码不是用例**：resume 不止重跑测试，而是用记录里的失败当 RED，走系统化调试
  找根因、TDD 最小改动修复。
- **断点续记**：若任务此前已收尾（`task.json` 被删），resume 会按原 `task_id/slug/goal`
  重建它，否则 `Stop` 钩子是 no-op，复现/修复的尝试不会被记成新轮次。
- **容错**：所有 trace hook 沿用 `session-start.ps1` 的契约——`SilentlyContinue`、包 try、
  **永远 exit 0**、stdin 空/损坏不写文件、`<task>.json` 先写临时文件再原子替换。

## 8. `brainstorm` 技能：浏览器实时脑图

手动运行 `/superharness:brainstorm <主题>` 时，启动一个**零依赖 Node 服务器**，在浏览器
打开实时脑图，边对话边画图。

### 8.1 组件与数据流

```mermaid
flowchart LR
    subgraph CC["Claude Code (终端)"]
        CLAUDE["Claude<br/>边问边更新结构"]
    end
    subgraph SRV["server.cjs (零依赖 Node)"]
        WATCH["文件监听 mindmap.json"]
        WS["WebSocket 推送"]
        HTTP["HTTP: 静态页 + /event + /mindmap.json"]
    end
    subgraph FILES["会话目录 .superharness/<session>"]
        SNAP["content/mindmap.json<br/>(全量快照)"]
        EVT["state/events<br/>(点击事件 JSONL · 每次推送清空)"]
        EDT["state/edits<br/>(label/note 编辑 + 提交 JSONL · 合并前不清)"]
    end
    BROWSER["浏览器脑图<br/>mindmap.html + layout.js"]

    CLAUDE -->|写快照| SNAP
    SNAP --> WATCH
    WATCH --> WS
    WS -->|推送快照| BROWSER
    BROWSER -->|节点点击 POST /event| HTTP
    BROWSER -->|双击改文字/提交 POST /event| HTTP
    HTTP -->|node:click → 落盘| EVT
    HTTP -->|node:edit / submit → 落盘| EDT
    EVT -->|下一轮读取| CLAUDE
    EDT -->|提交后并入快照| CLAUDE
```

服务器对 `POST /event` 按消息 `type` 分流：`node:click` 写 `state/events`（每次推送
新快照时被清空，所以待处理的点击总是对应当前屏幕）；`node:edit` 与 `submit` 写
`state/edits`（**快照推送不清它**，持续到 Claude 合并后才清），两条管道互不干扰。

### 8.2 一次交互的时序

```mermaid
sequenceDiagram
    participant U as 用户
    participant C as Claude(终端)
    participant S as server.cjs
    participant B as 浏览器脑图

    C->>S: 启动服务器 (start-server.ps1)
    S->>B: 打开页面，WS 连接后推首张快照
    C->>C: 提一个问题/确定一个决策
    C->>S: 写入 content/mindmap.json (新 rev)
    S->>S: 文件监听发现变化
    S->>B: WebSocket 推送最新快照
    B->>U: 渲染新节点(按类型着色/状态高亮)
    U->>B: 点击某节点表达选择
    B->>S: POST /event (node:click)
    S->>S: 追加到 state/events (JSONL)
    C->>S: 下一轮读取 events
    C->>U: 结合终端文字理解意图，继续推进
    Note over C,B: 结束后生成设计文档到 superharness/specs/
```

### 8.3 节点编辑：浏览器改文字、提交后并入设计

除了"点节点表达选择"，用户还能在浏览器里**直接编辑节点的 `label`/`note`**，把改动
**批量提交**回 Claude 并入设计——让脑图从"只读投影"变成"可回写的协作画布"。

**前端（`mindmap.html`）**：双击节点弹出编辑面板（label 输入框 + note 文本框），保存
后记入本地 `pendingEdits`（`id → {label, note}`），并 `POST /event` 发一条 `node:edit`；
顶栏「提交 (N)」按钮显示待提交条数，点击后发一条 `submit`。保存即**乐观更新**：界面立刻
显示新文字，但这层覆盖**一直保留到 Claude 合并后的权威快照到达**（在 WS 收到更高 `rev`
的快照时由 `clearPending()` 清除），提交按钮只锁定为「已提交，等待合并…」，绝不自己抹掉
覆盖，否则会在等待期把标签闪回旧值。

**协议（`state/edits` JSONL）**：

```json
{"type":"node:edit","id":"q1-a","label":"新标签","note":"新备注","timestamp":...}
{"type":"submit","timestamp":...}
```

只有 `label`/`note` 可编辑；同一 `id` 后写覆盖先写。

**Claude 侧的编辑轮（SKILL.md）**——这是一段**阻塞式握手**，关键在次序：

```mermaid
flowchart TD
    INV([Claude 邀请用户编辑节点文字]) --> BASE["①先建基线：清空 state/edits<br/>(防上一轮的陈旧 submit 立刻满足等待)"]
    BASE --> TELL["②提示：去浏览器双击改 label/note，逐个保存，改完点「提交」"]
    TELL --> WAIT["③不结束回合，阻塞等待<br/>Monitor 监听 state/edits 里的 submit 行<br/>(无 Monitor 则 ScheduleWakeup ≤60s 轮询)"]
    WAIT --> READ["④收到 submit：读 state/edits<br/>取所有 node:edit (同 id 后写胜)<br/>按 id 贴回当前快照树；已不存在的 id 忽略"]
    READ --> CONFLICT{与终端结论冲突?}
    CONFLICT -->|是| ASK["⑤在终端确认哪个为准"]
    CONFLICT -->|否| WRITE
    ASK --> WRITE["⑥改写 mindmap.json (rev+1)<br/>然后清空 state/edits"]
    WRITE --> DONE([权威快照推回浏览器，乐观覆盖被替换])
```

**为什么"先清基线再邀请"**：清空 `state/edits` 必须发生在邀请**之前**，否则上一轮残留
的 `submit` 行会让本轮的等待瞬间被满足（假提交）；同理还堵住了"用户手太快、submit 落在
还没开始 watch 的窗口"。`events` 与 `edits` 分两个文件、且 `edits` **不随快照推送被清**，
正是为了让乐观覆盖在整段等待期内稳定存在。

### 8.4 关键设计

- **消息协议**：Claude → 前端走"写全量快照文件"（含 `rev`/`status`/树形 `root`），
  服务器监听文件变化后 WebSocket 推送；前端 → Claude 走 `POST /event` 落盘 JSONL
  （`node:click` 进 `events`，`node:edit`/`submit` 进 `edits`），Claude 下一轮读取。
  两端通过**文件**解耦，无需复杂双向 RPC。
- **零依赖**：`server.cjs` 自己用 Node 原生 `http` + `crypto` 手写了 WebSocket
  握手与帧编解码（RFC 6455），不装任何 npm 包。
- **自动降级**：没有 Node 时流程退回纯终端脑暴，绝不阻塞。
- **自动退出**：空闲超时（默认 30 分钟）自动关闭，端口随机选取。
- **会话产物**：落在 `.superharness/`（已 `.gitignore`），不污染仓库。
